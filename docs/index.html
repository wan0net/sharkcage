GitHub Documentation
The Problem

AI agents run with your keys to the kingdom.

Permission prompts don't scale. Users click through them. Scanning tools find patterns but can't stop behaviour. Nothing enforces at the kernel level.

Full system access

AI agents inherit your user permissions. Every file, every network socket, every credential -- reachable by default.

Permission fatigue

Approve, approve, approve. After the 20th prompt, everyone reaches for --dangerously-skip-permissions.

Pattern matching, not enforcement

Static scanners catch known-bad strings. They can't stop a novel command or an unexpected network call at runtime.

No kernel-level enforcement

Other tools advise. Sharkcage enforces. The OS itself blocks the syscall before it completes. No userspace bypass.

How It Works

Five layers. Defence in depth.

Each layer is independent. Compromise one, the others still hold. Defence in depth, not defence in hope.

1

ASRT Sandbox Backend kernel

Registered with OpenClaw as the sandbox backend. Every call to run, read, or write is wrapped by srt. Per-session policies generated at startup and locked.

2

SOCKS5 Localhost Proxy network

Per-skill authentication tokens. Controls which localhost ports each skill can reach. Skills can't talk to each other's services or hit unexpected endpoints.

3

Security Scanner patterns

All detection patterns in one auditable JSON file. Blocks dangerous commands before they run. Redacts secrets and PII from outputs. Priority-ordered hooks.

4

Human Approval ai-invisible

The AI never sees the approval exchange. User approves in their own chat channel -- Signal, Telegram, whatever. No prompt injection vector. No social engineering surface.

5

Runtime Capability Discovery dynamic

No upfront manifests to maintain. The sandbox blocks what hasn't been approved. User grants capabilities as needed. Permissions grow from zero, not shrink from everything.

Platforms

Runs where you run.

Native kernel primitives on each platform. No containers. No virtualisation overhead. Just the OS doing what it already knows how to do.



macOS

Seatbelt sandbox profiles via sandbox-exec. No root required. Filesystem and network restrictions enforced by the XNU kernel.

Quick Start

One line. Running.

Installs sharkcage, OpenClaw, and srt. Sets up the sandbox. Adds sc to your PATH.

terminal 
# One-line install
curl -fsSL https://raw.githubusercontent.com/wan0net/sharkcage/main/install.sh | bash

# Setup wizard (configures OpenClaw + sandbox mode)
sc init

# Start everything
sc start
CLI Reference

The sc command.

Everything you need to manage sandboxes, skills, approvals, and audit logs from the terminal.

Command Description
sc init Initialize sharkcage configuration and generate sandbox profiles
sc start Start the supervisor and sandbox environment
sc stop Stop all running sandboxes and the supervisor
sc status Show running skills, sandbox state, and active policies
sc skill add <url> Install a skill from a repository URL
sc approve <skill> Review inferred capabilities and approve a skill's permissions
sc verify <path> Scan a skill directory for security issues before installation
sc audit View the full audit log of all tool calls and results
sc audit --blocked Filter audit log to show only blocked or denied calls
sc audit --skill <name> Filter audit log by skill name
sc config add-service <host> Add an allowed host to the outer sandbox network policy
sc sign <path> Ed25519 sign a skill manifest
sc trust <fingerprint> Trust a skill signer's public key
sc upgrade Safely upgrade OpenClaw with backup and rollback
