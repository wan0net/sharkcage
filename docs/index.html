The pitch

You have tasks. You have small computers. You yeet the tasks at the computers.

The computers run AI coding agents -- Claude Code, OpenCode, Aider -- autonomously. You go get coffee. Or sleep. Or touch grass. Whatever people do when they're not babysitting AI.

When you come back, there are draft PRs waiting for you. Reviewed, tested, ready to merge. Like a very cheap, very quiet development team that never asks about sprint velocity.

If something scary happens -- like the AI wanting to rm -rf / -- yeet pauses and texts you. Because even reckless abandonment has limits.

How it works
You --> yeet run --> Nomad --> Cheap PC --> AI Agent --> Draft PR --> You (hours later, caffeinated)

That's it. That's the architecture diagram. No, there isn't a second slide.

The yeet CLI dispatches a parameterized Nomad job. Nomad schedules it onto a machine that has the right project cloned and the right USB devices plugged in. A shell script (run-agent.sh) handles git, spawns the AI, commits results, opens a PR, and records cost. Nomad handles health, retries, logs, and fleet management. You handle being elsewhere.

The entire implementation is about 1,800 lines across a shell script, a TypeScript CLI, an HCL job template, and Ansible playbooks.

Features

Agent-agnostic

We don't play favorites. Claude Code, OpenCode, Aider -- whoever gets the job done. Bring your own agent if you want. We're not your mom.

Device-aware routing

Need a YubiKey plugged in? yeet knows which box has one. It's like Uber but for USB devices.

Cost controls

Set a budget so your AI doesn't spend your rent money on API calls. Capitalism requires guardrails.

Approval gates

If the AI tries something sketchy, yeet pauses and asks you first. Like a responsible adult.

Kernel-level sandboxing

Each task runs inside an NVIDIA OpenShell sandbox. Landlock, seccomp, network namespaces. The agent can only touch what the policy says.

Fleet management

Drain a runner, take it offline, bring it back. Like managing a very small, very cheap data center.

Nomad-powered

We literally just wrapped Nomad's API. Standing on the shoulders of HashiCorp giants. They did the hard part.

Adding a new runtime means adding a case branch to a shell script. That's the whole plugin system.

CLI
# Yeet a task and forget about it
yeet run peer6 "implement the notification system"

# Pick your weapon
yeet run peer6 "refactor auth" --runtime opencode --model anthropic/claude-sonnet-4
yeet run peer6 "refactor auth" --runtime claude --model opus

# Needs hardware? yeet routes to the right box.
yeet run login2 "test FIDO2 flow" --needs yubikey

# Check on your fleet of discount workstations
yeet runners

# What's happening right now
yeet status

# Oh no, what's it doing
yeet logs peer6-abc123

# It's 90% done, just fix the edge case
yeet continue peer6-abc123 "also handle the null case in auth"

# STOP STOP STOP
yeet stop peer6-abc123

# How much did that cost me
yeet cost --period week

Every command is a thin wrapper around a Nomad API call. If yeet doesn't do what you need, nomad CLI does. They're interchangeable.

Sandboxing

Letting an AI agent loose on a machine with full access is a bad idea. So we don't.

Every task runs inside an NVIDIA OpenShell sandbox -- the new hotness in agent isolation. The standalone openshell-sandbox binary wraps the agent process in kernel-level isolation -- no Kubernetes, no Docker, no overhead. Just Linux security primitives doing what they were designed to do.

Landlock LSM

Kernel-enforced filesystem allowlists. The agent can read/write its project workspace and /tmp. Everything else is invisible.

seccomp BPF

Syscall filtering. No privilege escalation, no raw sockets, no kernel module loading. The boring kind of security.

Network namespaces

All traffic forced through an HTTP proxy with per-binary OPA policies. The agent can reach the LLM API and GitHub. Nothing else.

Policies are generated per-task from the dispatch metadata. A review task gets a read-only workspace. A task that needs a YubiKey gets /dev/yubikey-1 in its allowlist and nothing else in /dev/. A task that doesn't declare a device can't touch one, even on a node that has one plugged in.

# What a sandbox policy looks like
filesystem:
  read_only:  [/usr, /lib, /etc, /opt/yeet/devices/]
  read_write: [/opt/yeet/workspaces/peer6/, /tmp]
network:
  allowed_hosts:
    - api.anthropic.com:443
    - github.com:443
    - registry.npmjs.org:443

The agent physically cannot read your SSH keys, call your internal APIs, or phone home to anywhere not on the list. Defense in depth without the DevOps tax.
