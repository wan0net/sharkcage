sharkcage

OpenClaw inside a kernel-level sandbox. Per-skill ASRT sandboxing. Approve once, enforce always.

GitHub Documentation
The pitch

OpenClaw is great. But skills run as untrusted code with no permission model. The sandbox is global. Eventually everyone ends up at --dangerously-skip-permissions.

Sharkcage fixes this. The entire OpenClaw binary runs inside a kernel-level outer sandbox. Every skill gets its own inner sandbox, scoped to exactly what you approved. No permission prompts at runtime. No prompt fatigue. No skipping.

Approve once at install. The kernel enforces it from then on.

How it works
sc start --> Supervisor --> Outer ASRT --> OpenClaw --> IPC --> Inner ASRT --> Skill

The supervisor is the only unsandboxed process (~200 lines). It owns all ASRT sandbox lifecycles. OpenClaw talks to skills via IPC through the supervisor — they never touch each other's network scope or filesystem.

# Two layers of kernel enforcement:
Outer ASRT → OpenClaw (entire binary, init-locked, signed)
  network: [signal-cli, openrouter.ai]
  deny: [~/.ssh, ~/.aws, ~/.gnupg]

Inner ASRT → meals skill (scoped to approved capabilities)
  network: [meals-api.wan0.cloud]
  filesystem: none

Inner ASRT → HA skill
  network: [homeassistant.local:8123]
  filesystem: none
Features

Entire process sandboxed

The outer ASRT contains the whole OpenClaw binary — not just plugins. Even a vulnerability in OpenClaw itself can't escape the outer sandbox constraints.

Per-skill isolation

Each skill runs in its own ASRT sandbox. Meals can't reach HA. HA can't reach meals. Installing a new skill never widens the gateway's attack surface.

Approve once, enforce always

Review capabilities at install time. The kernel enforces them from then on. No prompts during use. No fatigue. No skipping.

AI capability inference

Existing ClawHub skills have no manifests. Sharkcage reads the skill code and generates one automatically. User reviews and approves. Skill runs sandboxed.

Init-locked gateway

The outer sandbox config is generated at setup and signed. Widening it requires deliberate action, re-signing, and restart. Tampering → won't start.

Audit trail

Every tool call logged: timestamp, tool, args, result, skill, capability. Blocked calls logged too. Queryable from the CLI or dashboard.

The permission problem

Before sharkcage

  • Skills run as untrusted code
  • Global sandbox or per-action prompts
  • Prompt fatigue → skip permissions
  • No per-skill network scope
  • Malicious skill reaches anything

With sharkcage

  • Entire OpenClaw binary in outer ASRT
  • Each skill in its own inner ASRT
  • Approve once at install
  • Kernel-enforced per-skill network scope
  • Violations blocked and logged silently
CLI
# Start everything
sc start

# Install a skill
sc plugin add https://github.com/wan0net/sharkcage-skill-meals

# Review and approve capabilities
sc approve meals

# Scan a skill for issues
sc verify ./my-skill

# Check the audit log
sc audit --skill meals
sc audit --blocked

# Add a host to the outer sandbox
sc config add-service api.telegram.org

# Stop everything
sc stop
Sandboxing

Sharkcage uses the Anthropic Sandbox Runtime (ASRT) — OS-level primitives, no containers, no overhead.

macOS: Seatbelt

sandbox-exec wraps the process. Filesystem and network restrictions enforced by the macOS kernel. No root required.

Linux: bubblewrap + seccomp

Bind-mount filesystem namespaces. Network namespace with egress proxy. seccomp BPF syscall filtering.

Silent violations

Blocked calls get ECONNREFUSED or EPERM. The skill doesn't know why. The audit log does. No prompts.
